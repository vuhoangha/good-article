Swift in React Native - The Ultimate Guide Part 2: UI Components

Andrei Pfeiffer
May 4, 2018

This is a 2 part series intended to help all web & mobile developers that need to build custom Swift UI Components and use them in React Native applications.

This second part focuses on UI Components, while part 1 covers native Modules.

You can check out the github repo if you want to skip directly to the code.

Update 04 November 2018:
This guide uses React Native v0.57, Swift 4.2 and Xcode 10. If you have different versions, newer or older, some aspects might not work as expected.

During this guide, you will implement:

  • Two Counter components: one in React Native, the other in Swift.
    Tapping them increases each counter individually.
  • Communication between the 2 components.
    Long pressing on one counter will update the other one.

The main topics covered in this article include:

  1. Setup
  2. How to expose a Swift UI Component to JS
  3. How to send props to a Swift Component
  4. How to expose a Component Event Emitter
  5. How to expose methods on the ViewManager
  6. How to extract your React Native Component

Before implementing any type of communication, let’s create our UI component, expose it to Obj-C, then expose it to React Native and access it from React.

1. Setup

Create a new project

If you already have a React Native project, you can skip this step.

If you don’t have a project yet, you should create a new one. Please follow the official Getting Started guide of Building Projects with Native Code (not the Quick Start).

Throughout this guide, you will build a Counter App, so let’s create a new project, in case you haven’t got one already from part 1:

react-native init CounterApp
cd CounterApp
react-native run-ios

Now, let’s clean up the App component and implement a simple, but good looking counter:

// App.js
...
import {
  ...
  TouchableOpacity
} from 'react-native'
export default class App extends Component {
  state = {
    count: 1
  };
  increment = () => {
    this.setState({ count: this.state.count + 1 })
  }
  render() {
    return (
      <View style={styles.container}>
        <TouchableOpacity
          style={[styles.wrapper, styles.border]}
          onPress={this.increment}
        >
          <Text style={styles.button}>
            {this.state.count}
          </Text>
        </TouchableOpacity>
      </View>
    );
  }
}
const styles = StyleSheet.create({
  container: {
    flex: 1, alignItems: "stretch"
  },
  wrapper: {
    flex: 1, alignItems: "center", justifyContent: "center"
  },
  border: {
    borderColor: "#eee", borderBottomWidth: 1
  },
  button: {
    fontSize: 50, color: "orange"
  }
});

Your app should have this amazing look and increment the counter when tapped:

Next, open ios/CounterApp.xcodeproj in Xcode.

2. How to expose a Swift UI Component to JS

First of all, exposing your custom View requires a tiny extra step, compared to Modules. You won’t actually expose your View directly. Instead, you will expose a ViewManager that will return an instance of your custom View.

Note: it’s not required to have 2 separate files, but it’s a good practice. You can put your entire View logic inside the ViewManager, if you really want to.

Create a Swift ViewManager

From Xcode, just go to:

  • File → New → File… (or CMD+N)
  • Select Swift File
  • Name your file CounterViewManager
  • In the Group dropdown, make sure to select the group CounterApp, not the project itself.

Configure the Objective-C Bridging Header

After you create the Swift file, you should be prompted to choose if you want to configure an Objective-C Bridging Header. Select “Create Bridging Header”.

This file is usually named YourProject-Bridging-Header.h. Don’t change this name manually, because Xcode configures the project with this exact filename, as you can see in the Build Settings tab:

Note: there is only one Bridging Header per project, so once you have configured it, you won’t be prompted to do it again.

Let’s add 2 of the headers we will need right away, in case you don’t have them already:

// CounterApp-Bridging-Header.h
...
#import "React/RCTBridgeModule.h"
#import "React/RCTViewManager.h"

To start with, let’s create a basic text label in your newly created Swift file:

// CounterViewManager.swift
@objc(CounterViewManager)
class CounterViewManager: RCTViewManager {
  override func view() -> UIView! {
    let label = UILabel()
    label.text = "Swift Counter"
    label.textAlignment = .center
    return label
  }
}

Create the React Native Bridge

You will need a new Obj-C file as well:

  • File → New → File… (or CMD+N)
  • Select Objective-C File
  • Name your file CounterViewManager
// CounterViewManager.m
#import "React/RCTViewManager.h"
@interface RCT_EXTERN_MODULE(CounterViewManager, RCTViewManager)
@end

Access your Component from JS

Now, you should have access to your native View from JS, using the internal React Native module requireNativeComponent.

Important: notice that we have exposed our native View as CounterViewManager, but require it in JS as CounterView. If the exposed name ends with Manager, React Native will automatically remove the Manager ending.

For this tutorial, let’s add the CounterView component before closing the container view:

// App.js
import {
  ...
  requireNativeComponent
} from 'react-native'
const CounterView = requireNativeComponent("CounterView")
...
render() {
  return (
    <View style={styles.container}>
      ...
      </TouchableOpacity>
      <CounterView />
    </View>
  )
}

And… you’ll see nothing. Your component does render actually. You can verify this by inspecting the native component tree in Xcode. But it has no dimensions, so you can’t see it.

So let’s add a few previously defined styles:

// App.js
...
render() {
  return (
    ...
      <CounterView style={ styles.wrapper } />
    ...
  )
}

Now you should be able to see your Swift view.

Main queue setup warning

You might get a warning telling you that you didn’t implement the requiresMainQueueSetup method.

To suppress this warning, you have to override this method on your Swift ViewManager, and return a Boolean:

  • true if you need this class initialized on the main thread
  • false if the class can be initialized on a background thread
// CounterViewManager.swift
class ... {
  ...
  override static func requiresMainQueueSetup() -> Bool {
    return true
  }
}

Create your separate Swift UI View

So far so good, your bridge works. Next, let’s create your native Swift UI View, separate from its ViewManager. You will see later why.

  • File → New → File… (or CMD+N)
  • Select Cocoa Touch Class
  • Name your file CounterView
  • Subclass from UIView

Now, let’s rebuild the React view in Swift:

// CounterView.swift
import UIKit
class CounterView: UIView {
  @objc var count = 0 {
    didSet {
      button.setTitle(String(describing: count), for: .normal)
    }
  }
  override init(frame: CGRect) {
    super.init(frame: frame)
    self.addSubview(button)
    increment()
  }
  required init?(coder aDecoder: NSCoder) {
    fatalError("init(coder:) has not been implemented")
  }
  lazy var button: UIButton = {
    let b = UIButton.init(type: UIButton.ButtonType.system)
    b.titleLabel?.font = UIFont.systemFont(ofSize: 50)
    b.autoresizingMask = [.flexibleWidth, .flexibleHeight]
    b.addTarget(
      self,
      action: #selector(increment),
      for: .touchUpInside
    )
    return b
  }()
  @objc func increment() {
    count += 1
  }
}
Important: we need to use the @objc directive on each method / property that needs to be called or accessed from Obj-C.

And return your newly created CounterView from the Manager:

// CounterViewManager.swift
@objc(MyViewManager)
class MyViewManager : RCTViewManager {
  override func view() -> UIView! {
    return CounterView()
  }
  ...
}

Now, you should be able to independently increment the 2 counters:

  • on top, the React counter in orange;
  • on the bottom, the Swift counter in blue;

Next, let’s focus on the communication between the two.

3. How to send props to a Swift Component

In React, it’s a common practice to send props to your child components. Sending them to a native component is very easy, as well.

All you need to do is to expose your View variables using the RCT_EXPORT_VIEW_PROPERTY Macro in your ViewManager:

// CounterViewManager.m
...
@interface ...
RCT_EXPORT_VIEW_PROPERTY(count, NSNumber)
@end

For this tutorial, let’s expose your count variable, because you already have it in your View and because you might want to initialize it with a value sent from React.

Important: you have to use Obj-C types for variables exposed to React Native.

Because the count variable was inferred to Int, which is a Swift type, you have to change its type to NSNumber and adjust your code a bit:

// CounterView.swift
...
  @objc var count: NSNumber = 0 {
    ...
  }
  @objc func increment() {
    count = count.intValue + 1 as NSNumber
  }
...

Now you should be able to pass your own count from React to the Swift component:

// App.js
...
  <CounterView
    style={ styles.wrapper }
    count={2}
  />
...
Important: types must be respected. For instance: if you have defined a property as NSNumber, you cannot pass it as a String, or vice-versa.

Keeping your variables private (optional)

The previous example worked because the count variable is public. If you make it private, you will get an error telling you that:

[CounterApp.CounterView setCount:]
unrecognized selector sent to instance

To overcome this issue, you can implement a public setCount method and update your private variable explicitly:

@objc func setCount(_ val: NSNumber) {
  count = val
}

4. How to expose a Component Event Emitter

So far, you have implemented a one-way communication, from React to Swift. Next, let’s implement a solution to allow the Swift component to call your React component, so that you can subscribe to events triggered on the native side:

First, you need an optional RCTDirectEventBlock method that will be called when you long press the button.

Note: there’s also another type of event, called RCTBubblingEventBlock.
Unfortunately, the difference between the two is not documented, but for most use-cases, you shouldn’t need it.
// CounterView.swift
class ... {
  ...
  @objc var onUpdate: RCTDirectEventBlock?
  lazy var button: UIButton = {
    ...
    let longPress = UILongPressGestureRecognizer(
      target: self,
      action: #selector(sendUpdate(_:))
    )
    b.addGestureRecognizer(longPress)
    return b
  }()
  @objc func sendUpdate(_ gesture: UILongPressGestureRecognizer) {
    if gesture.state == .began {
      if onUpdate != nil {
        onUpdate!(["count": count])
      }
    }
  }
  ...
}

If you have to send any data to a RCTDirectEventBlock method, you must return a [AnyHashable:Any] structure. This means that you can’t pass a String or Int directly, you have to put them in a Dictionary.

Next, you need to expose this method to React Native’s bridge:

// CounterViewManager.swift
...
RCT_EXPORT_VIEW_PROPERTY(count, NSNumber)
RCT_EXPORT_VIEW_PROPERTY(onUpdate, RCTDirectEventBlock)
...

And you also have to add the corresponding header in your Bridging-Header:

// CounterApp-Bridging-Header.h
...
#import "React/RCTEventEmitter.h"

Finally, you should be able to receive these events in React:

// App.js
...
class App extends Component {
  ...
  update = e => {
    this.setState({
      count: e.nativeEvent.count
    })
  }
  render() {
    return (
      ...
        <CounterView
          style={ styles.wrapper }
          count={2}
          onUpdate={this.update}
        />
      ...
    )
  }
}

Now, when you long-press the Swift counter, the React counter should update with the same value.

Important: any RCTBubblingEventBlock or RCTDirectEventBlock property must have the “on” prefix, otherwise it will NOT be triggered. For example:
- onUpdate is valid;
- update or OnUpdate are invalid.

5. How to expose methods on the ViewManager

So far, you’ve implemented a two-way communication from Swift to React and vice-versa, so what more should you need? In the vast majority of use cases, the previous methods should be enough.

But what if:

  • you need to call a specific method on the Swift Component, without passing any props?
  • or, you need to call a method with multiple arguments?

Using props alone to achieve this is tricky. So, let’s see how to update the Swift Component from React Native, without using props.

First, let’s implement a Swift update method that updates the count:

// CounterView.swift
class ... {
  ...
  @objc func update(value: NSNumber) {
    count = value
  }
}

In order to be able to call this method on a particular instance of CounterView, we need to identify the instance. We do this by querying the React Native bridge for a specific node, that we have to pass from our React code (we’ll see how in a moment):

// CounterViewManager.swift
class ... {
  ...
  // 1
  @objc func updateFromManager(_ node: NSNumber, count: NSNumber) {
    
    DispatchQueue.main.async {                                // 2
      let component = self.bridge.uiManager.view(             // 3
        forReactTag: node                                     // 4
      ) as! CounterView                                       // 5
      component.update(value: count)                          // 6
    }
  }
}

Let’s got through the above code step by step:

  1. you need to implement a method that will be exposed to the bridge, and that will receive a node argument (you will use findNodeHandle from React Native to get its value, but more on this later);
  2. next, you dispatch to the main queue, because you are currently running on a background queue;
  3. you use the bridge UI Manager to query for the component instance;
  4. by passing the node handle that you get from React;
  5. and down casting it to your UI component, so you can call its methods;
  6. finally, you make the method call, by passing it the required arguments;

To have access to the UI Manager, you must import its header:

// CounterApp-Bridging-Header.h
...
#import "React/RCTUIManager.h"

Next, you also need to expose this method to the bridge, to be able to call it from JS:

// CounterViewManager.m
...
RCT_EXTERN_METHOD(
  updateFromManager:(nonnull NSNumber *)node
  count:(nonnull NSNumber *)count
)
...
Note: if the above syntax looks weird, you might want to read some details on how RCT_EXTERN_METHOD works.

Now you should have access to this method from JS. First, let’s import 2 more modules from React Native:

// App.js
import {
  ...
  UIManager,
  findNodeHandle
} from "react-native";
...

Next, you need to get the ref of the component:

<CounterView
  ...
  ref={e => this.counterRef = e}
/>

Call a new method when long-pressing the button:

<TouchableOpacity
  ...
  onLongPress={this.updateNative}
>

And implement this new method:

class ... {
  ...
  updateNative = () => {
    UIManager.dispatchViewManagerCommand(
      findNodeHandle(this.counterRef),                     // 1
      UIManager["CounterView"].Commands.updateFromManager, // 2
      [this.state.count]                                   // 3
    );
  }
  ...
}

Here you are telling React Native’s UIManager to dispatch a command to the ViewManager:

  1. the first argument is the node handle that you need to pass to your native method from the ViewManager. You have to use findNodeHandle and pass it the reference of your component;
  2. the second is the method you have exposed on your ViewManager;
  3. the third and last one is just an array of arguments;
Note: this feature can be implemented much easier by updating the counter prop. I’ve used this method only to show how to implement method calls on the ViewManager.

6. How to extract your React Native Component

In the end, you might want to extract your React Native Component into a separate file, so you can easily reuse it across your application:

// CounterView.js
import React, { Component } from "react";
import PropTypes from "prop-types";
import {
  requireNativeComponent,
  UIManager,
  findNodeHandle
} from "react-native";
const COMPONENT_NAME = "CounterView";
const RNCounterView = requireNativeComponent(COMPONENT_NAME);
export default class CounterView extends Component {
  static propTypes = {
    count: PropTypes.number,
    onUpdate: PropTypes.func
  };
  _onUpdate = event => {
    if (this.props.onUpdate) {
      this.props.onUpdate(event.nativeEvent);
    }
  };
  render() {
    const { count, style } = this.props;
    return (
      <RNCounterView
        style={style}
        count={count}
        onUpdate={this._onUpdate}
        ref={ref => this.ref = ref}
      />
    );
  }
  update = (...args) => {
    UIManager.dispatchViewManagerCommand(
      findNodeHandle(this.ref),
      UIManager[COMPONENT_NAME].Commands.updateFromManager,
      [...args]
    );
  };
}

This way, you are hiding from your consumers all React Native integration and they can use your custom native View like a regular React component:

// App.js
import CounterView from './CounterView'
export default class App extends Component {
  
  ...
  update = e => {
    this.setState({ count: e.count })
  }
  updateNative = () => {
    this.counterRef.update(this.state.count);
  }
  ...
}
